AI 编程四个阶段的主要形式与痛点详解

阶段一 · 2021—2023
网页对话问答
ChatGPT / Claude.ai / 文心一言
工程师角色：提问者 + 执行者
主要使用形式
1
复制代码粘贴进聊天框
↓
2
描述问题 / 需求（往往语焉不详）
↓
3
拿到回答，手动复制粘贴回编辑器
↓
4
发现不对，再回去继续问（新对话从零开始）
典型场景：问"帮我写一个 Python 函数"、"解释这段报错"、"给我一个 SQL 查询"
核心痛点
零上下文：AI 不知道你的项目结构、命名规范、依赖版本，给出的代码要大量改造才能用
搬运摩擦：每次都要在浏览器和编辑器之间切换，复制粘贴本身就消耗大量注意力
对话无记忆：新开一个对话，之前的背景全部归零，同样的背景要反复解释
幻觉难识别：API 名字、函数签名经常编造，看起来合理但跑起来报错，新手很难甄别
只问单文件：涉及多文件交互的问题，靠手动粘贴几乎无法准确描述完整关系
核心矛盾 & 演化动力
矛盾本质
AI 生活在聊天框里，代码生活在编辑器里。两者之间没有桥梁，人成了唯一的"数据搬运工"。
推动了 IDE 插件方向：把 AI 搬进编辑器，消灭物理切换成本，让 AI 直接"看见"代码。
阶段二 · 2023—2024
IDE 插件 · 内联辅助
GitHub Copilot / Cursor / Continue / 通义灵码
工程师角色：提问者 + 执行者（少了搬运）
主要使用形式
1
Tab 补全：写一半，AI 自动续写当前行或整个函数
↓
2
内联 Chat：选中一段代码，直接问"解释这段 / 帮我重构"，结果贴在旁边
↓
3
侧边对话：当前文件 + 少量关联文件作为上下文，问更复杂的问题
↓
4
AI 给出 diff，人来 Accept / Reject，最终执行仍靠人
典型场景：写注释自动生成函数体、选中函数问"有没有 bug"、边写边接受 Tab 建议
核心痛点
单文件局限：上下文窗口有限，AI 只能看见当前文件，跨文件的调用关系、类型依赖无法感知
仍需人逐步执行：AI 给建议，人来改代码，人来跑测试，人来看报错——节奏仍是"建议→人动手"
Tab 依赖症：过度接受补全建议会让工程师对代码失去掌控感，"加了一段不知道为什么的代码"
大任务拆不开："帮我实现这个功能"这类任务，AI 只能给代码片段，工程师需要自己把片段拼起来
无法调用工具：AI 无法跑测试、看终端输出、调用命令行，只能生成静态文本
核心矛盾 & 演化动力
矛盾本质
AI 能看见代码了，但仍不能"动"代码。每一个改动、每一次测试、每一次修复，仍需要人来点击、执行、观察。
推动了 Agent 方向：让 AI 不只给建议，而是直接读写文件、跑命令、形成自主闭环。
阶段三 · 2024—2025
编程 Agent · 自主执行
Claude Code / Devin / Cursor Agent / GitHub Copilot Workspace
工程师角色：任务描述者 + 审核者
主要使用形式
1
自然语言描述目标："帮我给 user 表加一个 email 验证功能"
↓
2
Agent 自主探索代码库，读取相关文件，理解项目结构
↓
3
Agent 写代码 → 跑测试 → 读报错 → 自修复，多轮循环
↓
4
人审核 PR / diff，决定是否接受，而非逐行指挥
典型场景：让 Agent 实现整个功能模块、迁移框架版本、修复一批 Bug、写全套测试
核心痛点
安全漏洞激增：2025 年 6 月数据，AI 引入的安全漏洞每月超 1 万个，是年初的 10 倍
审计跟不上速度：代码生成速度远超人工审查能力，"写得快、查不完"
架构静默漂移：Agent 会选择旧依赖、绕过现有约定，几次迭代后代码库开始偏离既定架构
错误难以预测：失败不规律——同样的任务有时成功有时跑偏，调试成本转移给了审核环节
多 Agent 协同混乱：多个 Agent 并行工作时，冲突、重复修改、互相覆盖的问题无法用"告诉它别这样做"解决
核心矛盾 & 演化动力
矛盾本质
Agent 太快了。速度优势反而成了问题——生成的代码无法被同等速度地审查和验证，可靠性成了新瓶颈。
推动了 Harness 工程：不靠"告诉 AI 别犯错"，而是构建系统性约束，让错误从机制上不可能发生。
阶段四 · 2026 至今
Harness 工程 · 驾驭 Agent
AGENTS.md / Rules files / CI gates / PEV loops
工程师角色：系统设计者 + 掌舵人
主要使用形式
1
Rules files / AGENTS.md：把约束写成文件，Agent 启动时自动注入，告知边界、禁区、规范
↓
2
确定性 CI 门禁：Linter、安全扫描、类型检查全部设为 error 级，违规直接拦截 PR
↓
3
PEV 验证循环：Plan → Execute → Verify，每步结果自动检查，不通过则 Agent 自行重试
↓
4
工程师分析 Agent 犯错模式，将修复逻辑写入 Harness，而非事后人工修补
典型场景：写 AGENTS.md 定义代码规范、把安全规则接入 CI、设计 Agent 任务拆分策略
当前挑战
Harness 本身需要专业设计：设计约束系统要求深厚的系统工程经验，入门门槛比"写提示词"高得多
行业规范尚未成熟：AGENTS.md 等标准 2025 年 8 月才刚起步，最佳实践仍在快速演化中
工程师角色转型阵痛：从"写代码"到"设计写代码的系统"，思维方式需要重构，很多人还未完成转变
责任边界模糊：AI 写的代码出了问题，谁负责？如何在团队中建立共识、划定审核责任
过度依赖风险：当 Harness 运行良好，工程师对底层代码的掌控感会进一步降低，长期影响待观察
核心洞见
范式本质
从"我来写代码"→"我来设计让代码被可靠写出来的系统"。更好的 Harness 往往比更好的模型更重要——赢家是设计了最好 Agent 环境的团队，不是用了最贵模型的团队。
开放问题：这条演化曲线的下一步是什么？多 Agent 协作系统、AI 自我改进 Harness？仍在探索中。
